# Stride: Task Decomposer (Inline)

## Purpose

Break down large goals and initiatives into well-sized, dependency-ordered tasks. Analyze the goal's scope, identify natural task boundaries, estimate complexity, and produce a structured payload that matches the Stride API batch creation schema.

**Pi context:** Pi does not ship with native subagent dispatch. In sibling plugins this runs as an isolated subagent; on Pi you execute these instructions inline. The decomposition logic is identical.

## When to invoke

Invoke this skill when:

- A claimed task is a **goal** type
- A claimed task has **large** complexity and no child tasks
- A work estimate exceeds 25 hours

Per the decision matrix in `stride-subagent-workflow`, goals must be decomposed before implementation — never implement a goal directly.

## Inputs (from your current context)

- A goal description (title + optional details)
- Optionally Stride task metadata if decomposing an existing task

Use the codebase to inform your decomposition. Read `AGENTS.md` for project conventions and explore `lib/` and `test/` for existing implementations related to the goal.

## Constraints

- Target **1–3 hour tasks** (minimum 1 hour, maximum 8 hours)
- **No `key_file` overlap** between sibling tasks — each task owns its files
- **No more than 8 tasks per goal** — decompose further if you exceed this
- Use **array indices** `[0, 1, 2]` for dependencies within the goal (identifiers don't exist yet)
- **Never specify identifiers** — they are auto-generated by the Stride API
- Always set `needs_review: false` — humans promote tasks to Ready
- Each nested task needs **full specification** per `stride-creating-tasks` (title, type, complexity, priority, description, why, what, where_context, key_files, verification_steps, testing_strategy, acceptance_criteria, patterns_to_follow, pitfalls)
- Batch endpoint root key is **`"goals"`**, NOT `"tasks"` (most common mistake — causes 422)

## Output

Produce a JSON payload shaped exactly like the request body for `POST /api/tasks/batch`:

```json
{
  "goals": [
    {
      "title": "<goal title>",
      "type": "goal",
      "complexity": "large",
      "priority": "...",
      "description": "...",
      "why": "...",
      "what": "...",
      "where_context": "...",
      "acceptance_criteria": "...",
      "tasks": [
        { "title": "...", "type": "work", ... },
        { "title": "...", "type": "work", "dependencies": [0], ... }
      ]
    }
  ]
}
```

See `stride-creating-goals/SKILL.md` for the full batch format and `stride-creating-tasks/SKILL.md` for the per-task field requirements. Both of those skills are the canonical references — this skill only covers the *decomposition* logic, not the field schemas.

## Important constraints

- Do not interact with the Stride API from this skill — you produce the payload; the caller posts it
- Keep the decomposition tight: if 4 tasks do the job, don't pad to 6
- Flag any dependency on work **outside** this goal using existing task identifiers (e.g. `["W47"]`) rather than indices
